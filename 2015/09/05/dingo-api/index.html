发表于 2015-09-05

dingo/api wiki (未校正)

Dingo Api 包是为了给开发者提供一套可以轻松且快速开发出属于自己的 API 的工具。尽管这个包的目的是尽可能地灵活，但是它仍未能涵盖所有的情况，解决所有的问题。

安装

前提环境:

  • Laravel 5.1 + 或者 Lumen 5.1 +
  • PHP 5.5.9 +

你首先要在你的项目中修改 composer.json 文件，然后运行 composer update 来安装最新版本的包。

或者你可以在命令行中运行 compsoer require 命令。

此时的包仍处于开发阶段，因此它不是一个正式发布的版本。你需要在 dev 中添加 minimum-stability。

一旦包安装成功后，接下来就看你使用的框架了。

标准树 (Standards Tree)

有三种不同的树: x , prs 和 vnd 。如何使用标准树取决于你正在开发的项目。

  • 未注册树(x)主要用于局部和私有的环境。
  • 个人树(prs)主要用户不是商业部署的环境。
  • 厂商树(vnd)主要用于公开部署的环境。

使用个人树或者厂商树在技术上意味着 IANA。

响应

一个运作的 API 是能在接收请求，并且返回一个客户端一个易处理的响应。一个 API 通常会返回一个易于处理的格式，比如 JSON。有很多不同的方法来返回响应，这很大程度上取决你的 API 有多复杂，或者他将会变得多复杂。

最简单的返回一个易处理的响应的方法就是在你的控制器中直接简单地返回一个数组或者对象。并不是每一个对象都可以被正确的格式化，所以你需要确保它实现了 ArrayObject 或者 Illuminate\Support\Contracts\ArrayableInterface 接口。

在这个案例中，我们的 User 类继承了 Illuminate\Database\Eloquent\Model, 这意味着它可以被格式化成一个数组，所以我们可以简单地返回用户的集合，就是 User:all() 返回的结果。同样你可以用它返回一个用户的信息。

dingo 包会自动的格式化响应为 JSON 格式， 并设置 Content-Type 头为 application/json 。

错误和错误响应

构建 API 中处理错误的部分可以说是一大痛处。代替手动创建错误响应的方式是，你可以简单地抛出一个继承了 Symfony\Component\HttpKernel\Exception\HttpException 的异常，这样 API 将会自动的为你处理响应。

这是一个 Symfony 内置的异常列表。

举例来说，当你尝试去更新一条已经被其他用户更新过的记录的时候，你可能会抛出一个 ConflictHttpException 。

这时包将会自动捕获抛出的异常，并将会转换成以 JSON 的方式展现。HTTP 响应的状态码也会变为这个异常的状态码。一个 ConflictHttpException 将会导致一个 HTTP 409 的状态码，如果你没有改变默认的错误格式，那么他将会以 JSON 的格式展现。

转换器

转换器可以让你轻松地统一的将对象转换成一个数组。通过一个转换器，你可以转换整形和布尔型变量，也包括分页结果和嵌套关系等。

术语 (Terminology)

转换器(transformer) 在本章节中会被大量使用。值得一提的是下列术语在本章节中被用到的含义。

  • 转换层(transformation layer) 是准备和处理转换器的层。
  • 转换器是一个类，将接受原生数据，并按照设定的格式返回一个数组。转换器是在转换层中工作的。

使用转换器

有以下几种方法可以使用转换器

将一个类注册生成转换器

当你讲一个给定的类注册成转换器的时候，你将可以直接从你的路由中返回那个给定的类(假设它可以被转换成一个数组)，他将会自动的通过转换器的处理。对于简单的 API 来说，这真是极好的。你可以在路由中使用模型(model)直接作为返回值。

资源键
利用回调

Fractal 转换层允许你注册一个当资源创建后被触发的回调函数。这个回调函数也可以接受一个 League\Fractal\Resource\Item or League\Fractal\Resource\Collection 的实例作为他的第一个参数，League\Fractal\Manager 作为他的第二个参数。然后你可以将它使用在一些更复杂的层面的资源进行交互。

最明显的例子是当你想要为分页数据设置光标执行或是为每一个基础请求改变序列。

如果你没有使用参数传递资源键，你可以传入一个空的数组，并简单地传递回调作为第三个参数。

定制转换层

你应该需要更多的途径了解你的数据是如何转换的，这样你可以在包中更轻松的实现你的转换层。你需要创建一个类，它实现了 Dingo\Api\Contract\Transformer\Adapter 接口，并需要有 transform 方法。

这个 transform 方法是唯一必须的方法，你可以任意添加其他你需要的方法。 transform 方法是为了获得 $response, 然后把它连同 $transformer 给你的转换层。你的转换层应该返回一个 transform 方法返回的一个数组。当然，你可以在这个列中做了所有工作，这样你的转换层会显得非常简单。

$binding 参数在你的转换层中的绝大部分特性中非常有用。就像添加一个额外数据或者允许其他的开发者通过回调函数在你的转换层中进行交互。

$request 参数是当前被执行的 HTTP 请求，当你的转换层需要查询字符值或其他关联数据时非常有用。

认证

大多数的 API 在返回响应之前需要某种形式的认证。在一个请求通过认证和没有通过认证时一个响应可能会有所不同。

这个包允许你配置不同的认证提供者。当请求尝试认证时，每一个被打开的认证提供者都将会被执行。

配置认证提供者

在配置文件中默认只打开了 HTTP 基础认证。这里有这个包现在支持的认证提供者的一个列表。

  • HTTP Basic (Dingo\Api\Auth\Provider\Basic)
  • JSON Web Tokens (Dingo\Api\Auth\Provider\JWT)
  • OAuth 2.0 (Dingo\Api\Auth\Provider\OAuth2)

HTTP Basic

这个认证提供者是 Laravel 和 Lumen 的默认认证。你需要配置这个提供者在你的一个服务提供者中或者启动文件。第二个参数是用来认证时鉴定的字段。

JSON Web Tokens (JWT)

这个包使用的第三方整合的 JWT 认证。请参考 tymon/jwt-auth 的 Github 页面来查看更多安装细节和配置。

一旦你使用了这个包，那么你可以在 config/api.php 中配置，或者一个服务提供者亦或是启动文件中。

OAuth 2.0

这个包使用的是第三方整合的 OAuth 2.0. 你一样可以安装 league/oauth2-server，自己配置服务或者使用桥梁包(bridge package)， lucadegasperi/oauth2-server-laravel。

为了简单起见，本次指南将会假设你使用的是桥梁包(bridge package)。

一旦你使用了这个包，你可以在一个服务提供者中或者启动文件中配置。

或者添加到一个服务提供者。

用户和客户端解释器

根据你开启的认证授权，你可能不需要使用这两个解释器。举例来说，如果你只允许客户端通过 OAuth 2.0 认证，那么你不需要设置一个用户的解释器。

解释器接受用户或者客户端的ID和需要使用这个ID返回一个实例给用户活客户端。这通常体现在为用户或者客户端查询数据库时。

自定义认证提供者

如果你正在开发一个遗留的系统或者需要其他形式的认证，你可以实现自己的认证提供者。

你的认证提供者应该实现 Dingo\Api\Contract\Auth\Provider 。 如果认证成功，你的提供者应该返回一个认证通过用户的实例。如果认证失败，应该抛出一个异常 Symfony\Component\HttpKernel\Exception\UnauthorizedHttpException.

一旦你使用的自己的认证提供者，那你应该在 config/api.php 中配置一下。

或者在启动文件亦或是在服务提供者。

频率限制

频率限制(节流)允许你限制统一客户端在给定时间内的请求次数。一个限制和其到期时间被定义为节流。这个包中默认有两个节流，一个已认证的节流和一个未认证的节流。

更改频率限制的键

频率限制默认情况下限制客户端的 IP 地址。要改别这个默认的行为，你可以注册自己的解释器，它应该返回一个被用于频率限制的字符串。

如果你只想为某些路由或者一组路由节流，你可以使用 limit 和 expires 配置你的路由。

这将为这个特定路由建立一个5分钟最多100次请求的限制。如果你将它设置在路由组上，那么其中的每一个路由都会有100得限制。

一个用户在每次限制中可以访问 /user 路由和 /posts 路由100次，这个限制不会在应用组中生效，但会在组里每个路由上生效。

资源

控制器通常可以表示为一个资源。资源可以表示为包含很多以 HTTP 动词命名的行为。

为了定义一个资源，我们可以使用 @Resource 注释。

第一个参数可以是资源的标示符或者是它的名字。我们也可以给定一个基础 URI 。

你也可以在注释的前面协商一段描述。

Actions

一个行为可以被表示为控制器中的一个路由方法。

你可以在 PHPDoc 中用一段语言来描述你的行为。

@Get, @Post, @Put, @Patch, @Delete

每一个行为可以用一个 HTTP 动词表示。你必须提供一个 URI 作为注释的第一个参数。

@Versions

一个行为可以再多个 API 版本中生效。当你要生成文档时，这个注释将确定那些行为将会被包含。

@Request

一个行为应该定义一个请求，可以被执行并能返回一个成功或者是白的响应。

一个请求应该包含一个主体。根据请求的类型，主体将会变化。对于 POST 请求，你应该使用一个字符串，然而你也需要设置它的内容类型。

如果你发送 JSON,你可以使用注释数组，他会自动的转换成 JSON 字符串。内容类型将会默认为 application/json。

你也可以包含额外的头信息。

如果你的行为根据不同的请求而响应不同，你可以认证请求。

@Response

一个 @Request 应该永远跟着一个 @Response, 他定义了状态码和内容类型，主体和头信息。

就像一个请求一样，响应的主体可以使字符串(请确保更改了内容类型)或者将会被转换成 JSON 的注释数组。

就像一个请求一样，你也可以包含头信息。

@Transaction

一个事务可以让你定义更多的请求和请求的响应。响应必须跟着请求，也可以为一个请求定义多个响应。

@Parameters

如果你的 URI 包含查询字符串参数，你可以在资源等级或者行为等级中定义他们。如果一个参数被定义在资源等级上，你也需要将它定义到每个行为或者资源上。

你也可以定义参数 type 以及他是否 required。
