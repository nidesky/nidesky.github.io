发表于 2015-09-05

dingo/api wiki

Dingo Api 包是为了给开发者提供一套可以轻松且快速开发出属于自己的 API 的工具。尽管这个包的目的是尽可能地灵活，但是它仍未能涵盖所有的情况，解决所有的问题。

安装

前提环境:

  • Laravel 5.1 + 或者 Lumen 5.1 +
  • PHP 5.5.9 +

你首先要在你的项目中修改 composer.json 文件，然后运行 composer update 来安装最新版本的包。

或者你可以在命令行中运行 compsoer require 命令。

此时的包仍处于开发阶段，因此它不是一个正式发布的版本。你需要在 dev 中添加 minimum-stability。

一旦包安装成功后，接下来就看你使用的框架了。

Laravel

打开 config/app.php , 并注册服务提供者(Service Provider)。

如果你想修改包配置文件中的配置，那么你可以尝试通过 Artisan command 发布它：

Lumen

打开 bootstrap/app.php , 然后注册服务提供者(Service provider)。

Facades

在包中有两个门面。你可以添加任何一个你想要的门面。

这是一个调度的门面，但是，他也在包中提供了辅助方法等方法。

这是一个针对于 API 路由提供者和可被用于获取当前路由，请求，检查当前路由名称的一个门面。

配置

大部分的包可以通过预先配置来尽可能快的构建 API。你可以使用 .env 文件来配置大多数的包，然而一些更详细地配置需要你要么发布配置文件(Laravel)，或者配置 bootstrap/app.php (Lumen)。你也可以通过使用 AppServiceProvider 的 boot 方法。

Important:如果你是用 Laravel 5，你必须发布你的配置文件或者一个服务提供商。不要使用 bootstrap/app.php 文件。

如果你使用 Laravel ，你可以通过 Artisan command 来发布配置文件：

标准树 (Standards Tree)

有三种不同的树: x , prs 和 vnd 。如何使用标准树取决于你正在开发的项目。

  • 未注册树(x)主要用于局部和私有的环境。
  • 个人树(prs)主要用户不是商业部署的环境。
  • 厂商树(vnd)主要用于公开部署的环境。

使用个人树或者厂商树在技术上意味着 IANA。

你可以在你的 .env 文件里配置。

子类型 (Subtype)

你的子类型通常是你的应用或项目全部小写的简称。
你可以在你的 .env 文件里配置。

前缀和子域名 (Prefixes and Subdomains)

如果你曾经开发过 API， 那么你一定会知道大部分的 API 不是放在一个子域名就是带有前缀。一个前缀或者子域名是必须的，但只需要其中之一即可。避免将版本号作为前缀或子域版本通过 Accept 报头处理。

你可以在你的 .env 文件里配置。

或者可以使用一个子域名。

版本 (Version)

这个版本是你的 API 的默认版本，它被用在每当版本没有提供几种情况下的回退。这个版本也会在生成 API 文档被当成默认的版本。

你可以在你的 .env 文件里配置

名称 (Name)

这个名字只会被用在当你使用 API Blueprint 命令生成文档的时候。这个名字会被用作生成文档的默认名称。

你可以在你的 .env 文件里配置

条件请求 (Conditional Requests)

条件请求默认开启，因为它会利用缓存功能在客户端可以缓存 API 请求的时候。
你可以在你的 .env 文件里配置

严格模式 (Strict Mode)

严格模式将会发送 Accept 头来代替默认的配置文件中指定的配置。这意味着你将不能通过你的浏览器访问你的 API 。

如果严格模式开启并且 API 应用了有效地 Accept 头，这将会抛出一个未处理的 Symfony\Component\HttpKernel\Exception\BadRequestHttpException ，你应当妥善处理这个异常。

你可以在你的 .env 文件里配置

认证提供者(Authentication Providers)

默认只开启 basic 认证。在后面的章节会涉及到更多认证相关的细节。

对于更复杂写的配置，你要添加一个服务提供者或者启动文件。

节流 / 访问频率限制(Throttling / Rate Limiting)

访问频率限制默认是关闭的。你可以注册自定义的带有频率限制的节流或者使用已经存在的认证状态或者未认证状态的节流。

对于更复杂写的配置，你要添加一个服务提供者或者启动文件。

响应转换器(Response Transformer)

Fractal 是默认的相应转换器。

当然，你可以配置你的 .env 文件。对于更复杂写的配置，你要添加一个服务提供者或者启动文件。

响应格式(Response Formats)

默认的响应格式是 JSON，JSON 响应格式被注册为默认的响应格式。

你可以在 .env 文件中配置默认的响应格式。更多的响应格式配置需要配置以下内容。

错误格式(Error Format)

当包发生错误的时候，他将会尝试生成一个通用的错误响应，而不是将异常展示给用户。你可以通过自己的喜好来配置错误的格式。

你必须在已发布的配置文件或者引导文件中进行配置。

调试模式(Debug Mode)

如果开启调试模式的话，包处理一般的错误时会显示详细的错误堆栈信息。

你可以配置你的 .env 文件。

响应

一个运作的 API 是能在接收请求，并且返回一个客户端一个易处理的响应。一个 API 通常会返回一个易于处理的格式，比如 JSON。有很多不同的方法来返回响应，这很大程度上取决你的 API 有多复杂，或者他将会变得多复杂。

最简单的返回一个易处理的响应的方法就是在你的控制器中直接简单地返回一个数组或者对象。并不是每一个对象都可以被正确的格式化，所以你需要确保它实现了 ArrayObject 或者 Illuminate\Support\Contracts\ArrayableInterface 接口。

在这个案例中，我们的 User 类继承了 Illuminate\Database\Eloquent\Model, 这意味着它可以被格式化成一个数组，所以我们可以简单地返回用户的集合，就是 User:all() 返回的结果。同样你可以用它返回一个用户的信息。

dingo 包会自动的格式化响应为 JSON 格式， 并设置 Content-Type 头为 application/json 。

响应生成器

响应生成器提供了一个流畅的接口来让我们更轻松构建一个更加个性化的响应。响应生成器一般和转换器一起使用。

设置响应状态

自定义响应格式

在配置那一章节中，我们简要的介绍了相应格式。默认情况下，包将自动设置使用 JSON ，并设置适当的 Content-Type 头。除了 JSON 格式外，还有 JSONP 格式。这个格式将响应包装在一个回调里面。你可以在你的 配置文件或者启动文件中通过简单地替换来注册这种格式。

默认情况下这个回调函数的参数是一个回调函，这可以通过传递的第一个参数来改变为类构造函数。如果查询字符串不包含一个和你回调函数名字相同的参数，那么将会默认返回一个 JSON 响应。

如果有需要，你也可以注册和使用自己的格式。你的格式应该继承 Dingo\Api\Http\Response\Format\Format。有以下方法应该被定义：formatEloquentModel, formatEloquentCollection, formatArray, getContentType。请查看抽象类来了解每个方法应该做什么，或者看一下已经定义的那些格式的类。

错误和错误响应

构建 API 中处理错误的部分可以说是一大痛处。代替手动创建错误响应的方式是，你可以简单地抛出一个继承了 Symfony\Component\HttpKernel\Exception\HttpException 的异常，这样 API 将会自动的为你处理响应。

这是一个 Symfony 内置的异常列表。

举例来说，当你尝试去更新一条已经被其他用户更新过的记录的时候，你可能会抛出一个 ConflictHttpException 。

这时包将会自动捕获抛出的异常，并将会转换成以 JSON 的方式展现。HTTP 响应的状态码也会变为这个异常的状态码。一个 ConflictHttpException 将会导致一个 HTTP 409 的状态码，如果你没有改变默认的错误格式，那么他将会以 JSON 的格式展现。

表单请求

如果你使用的是表单请求，那么你需要继承 API 表单请求的基类，或者你自己实现它。 API 表单请求的基类将会检查看看传入的请求是不是针对 API 的， 如果是的话，那么如果验证失败，他会抛出一个异常 Dingo\Api\Exception\ValidationHttpException。

这个异常将会被包正确的呈现，并会返回错误响应。

如果你更喜欢实现你自己的表单请求，你必须重载 failedValidation 和 failedAuthorization 方法。这些方法必须抛出一个上面提到的异常，而没有 Laravel 抛出的 HTTP 异常响应。
